# Agent-OS 图谱可视化设计：项目维度 vs 任务维度

> 基于 B 端 Agent Native 产品实践的可视化设计思考

---

## 一、两个维度的定位差异

| 维度 | 定位 | 用户视角 | 时间粒度 | 核心问题 |
|------|------|---------|---------|---------|
| **项目图谱** | 宏观治理层 | 项目经理 / 部门负责人 | 天/周/月 | 资源怎么分、风险在哪、谁在忙 |
| **任务图谱** | 微观执行层 | 业务执行者 / Agent 调度者 | 分钟/小时 | 任务走到哪一步了、谁卡住了、上下文在哪 |

**一句话区分**：
- 项目图谱 = **俯瞰战场**（兵力部署、战损、补给线）
- 任务图谱 = **望远镜看单兵**（这个士兵在干嘛、枪里几颗子弹、要不要增援）

---

## 二、项目维度图谱：应该展示什么

### 2.1 核心目标
让用户一眼回答三个问题：
1. 我的项目健康吗？（进度、风险、阻塞）
2. 我的数字员工在忙什么？（负载、效率、闲置）
3. 项目之间有什么依赖？（资源共享、阻塞传递）

### 2.2 节点类型

```
┌─────────────────────────────────────────────────────────┐
│                      项目维度节点                         │
├─────────────────────────────────────────────────────────┤
│  🏢 项目节点（中心）                                      │
│     - 项目名称、状态（健康/警告/危险）                     │
│     - 进度百分比、截止日期                                │
│     - 负责人（人类）                                      │
│                                                          │
│  🤖 Agent 节点（第一层）                                  │
│     - Agent 名称、当前状态（忙碌/空闲/异常）               │
│     - 本周调用次数、Token 消耗                            │
│     - 负载率（当前任务数 / 最大并发）                      │
│                                                          │
│  🛠️ Skill 节点（挂载在 Agent 下）                         │
│     - Skill 名称、调用频次                                │
│     - 成功率、平均耗时                                    │
│                                                          │
│  📁 资产节点（文件/洞察/产出）                             │
│     - 关键产出物（报告、图表、决策）                       │
│     - 关联文件数量                                        │
│                                                          │
│  ⚠️ 风险节点（动态浮现）                                  │
│     - 阻塞任务、逾期风险、异常告警                        │
│     - 自动计算：红色 = 高风险，黄色 = 警告                 │
└─────────────────────────────────────────────────────────┘
```

### 2.3 边的类型

| 边类型 | 含义 | 视觉表现 |
|--------|------|---------|
| **归属边** | Agent → 项目 | 实线，细 |
| **挂载边** | Skill → Agent | 虚线，淡色 |
| **依赖边** | 项目 A → 项目 B | 带箭头的粗线，红色 = 阻塞 |
| **产出边** | Agent → 资产 | 虚线箭头，指向产出物 |
| **负载边** | 人类 → Agent（审批关系）| 点状线 |

### 2.4 交互设计

- **悬停项目节点**：展开项目卡片（进度条、负责人、最近更新、风险项）
- **悬停 Agent 节点**：显示实时负载仪表盘（CPU-like 圆环）
- **点击风险节点**：右侧审计面板自动定位到异常 Session
- **双击空白处**：展开/收起所有子节点（鱼眼效果）

### 2.5 动态效果建议

- Agent 忙碌时节点轻微脉动（绿色 → 橙色）
- 风险出现时节点从边缘向中心"涟漪"扩散
- 新产出物生成时，从 Agent 飞向资产节点的粒子动画

## 三、任务维度图谱：应该展示什么

### 3.1 核心目标
让用户一眼回答三个问题：
1. 任务走到哪一步了？（状态机进度）
2. 谁在干什么？（Human / Agent / Skill 的实时分工）
3. 上下文是什么？（输入数据、中间产物、审批记录）

### 3.2 节点类型

```
┌─────────────────────────────────────────────────────────┐
│                      任务维度节点                         │
├─────────────────────────────────────────────────────────┤
│  🎯 目标节点（入口）                                      │
│     - 用户原始指令（自然语言）                            │
│     - 拆解后的子目标列表                                  │
│                                                          │
│  🧠 调度节点（群主 Agent）                                │
│     - 任务拆解计划（Planning）                            │
│     - 分配决策：哪个 Agent 干什么                         │
│                                                          │
│  👤 人类节点                                              │
│     - 审批节点（菱形）                                    │
│     - 输入/修正节点                                       │
│     - 显示：审批状态（待审批 / 已通过 / 已驳回）          │
│                                                          │
│  🤖 Agent 执行节点                                        │
│     - 具体 Agent 名称 + 执行动作                          │
│     - 开始时间、预计耗时、实际耗时                        │
│     - 状态：排队中 / 执行中 / 成功 / 失败 / 降级         │
│                                                          │
│  🛠️ Skill 调用节点                                        │
│     - 具体 Skill（如"图表生成"）                          │
│     - 输入参数、输出结果预览                              │
│     - Token 消耗                                          │
│                                                          │
│  📦 产物节点                                              │
│     - 中间产物（清洗后的数据、SQL 结果）                  │
│     - 最终产物（报告、图表、邮件）                        │
│     - 可点击预览                                          │
│                                                          │
│  ⚠️ 异常节点                                              │
│     - Agent 失败 / Skill 超时 / 数据异常                  │
│     - 自动降级路径（备选方案）                            │
│     - 人工介入入口                                        │
└─────────────────────────────────────────────────────────┘
```

### 3.3 边的类型

| 边类型 | 含义 | 视觉表现 |
|--------|------|---------|
| **顺序边** | 步骤 1 → 步骤 2 | 粗箭头，按时间流向 |
| **调度边** | 群主 Agent → 子 Agent | 虚线，带"分配"标签 |
| **审批边** | Agent → 人类 → Agent | 双向箭头，审批中闪烁 |
| **依赖边** | 产物 A → 步骤 B（输入）| 细实线，标注数据类型 |
| **异常边** | 失败步骤 → 降级步骤 | 红色虚线，带⚠️图标 |

### 3.4 任务图谱的两种视图模式

#### 模式 A：横向流程图（Flow View）
适合：看整体进度、审批节点、 bottlenecks

```
[用户目标] → [群主规划] → [Agent A 执行] → [👤 审批] → [Agent B 执行] → [产物]
                ↓              ↓                    ↓
           [子目标1]      [Skill调用]          [Skill调用]
           [子目标2]
```

#### 模式 B：纵向时间线（Timeline View）
适合：看执行历史、回查问题、审计追踪

```
10:00  [用户] 提出目标
10:01  [群主Agent] 拆解为3个子任务
10:02  [财报Agent] 开始执行 → 调用「数据提取」Skill
10:03  [财报Agent] 产出 Q3 营收表
10:04  [👤 用户] 审批通过
10:05  [图表Agent] 开始执行 → 调用「图表生成」Skill
10:06  [图表Agent] 产出柱状图 ✅
```

### 3.5 交互设计

- **点击执行中节点**：右侧展开实时日志（stdout 风格，滚动更新）
- **点击产物节点**：右侧预览产物内容（图表放大、表格展开）
- **点击异常节点**：显示错误详情 + 备选方案按钮（"重试"/"跳过"/"人工处理"）
- **拖拽节点**：重新编排任务顺序（仅调度层支持）

## 四、两个维度的联动关系

```
                    ┌──────────────┐
                    │   项目图谱    │  ← 宏观：看战场
                    │  （治理层）   │
                    └──────┬───────┘
                           │ 点击项目
                           ▼
                    ┌──────────────┐
                    │   任务列表    │
                    │  （项目下）   │
                    └──────┬───────┘
                           │ 点击任务
                           ▼
                    ┌──────────────┐
                    │   任务图谱    │  ← 微观：看单兵
                    │  （执行层）   │
                    └──────────────┘
```

**联动规则**：
- 项目图谱中 Agent 的"忙碌"状态 = 该 Agent 在任务图谱中有"执行中"节点
- 任务图谱中产物节点的数量 = 项目图谱中资产节点的更新来源
- 任务图谱中的异常节点 = 项目图谱中风险节点的生成来源

## 五、B 端用户真正需要的 6 个可视化能力

基于你文档里的 Agent Native 设计原则，B 端用户需要的不是"好看"，而是**可信任、可控制、可审计**。

| 能力 | 项目维度体现 | 任务维度体现 |
|------|------------|------------|
| **1. 实时状态感知** | 所有项目的健康红绿灯 | 每个任务的执行进度条 |
| **2. 责任归属清晰** | 谁负责哪个项目、哪个 Agent | 每一步是谁干的（人还是 Agent）|
| **3. 异常快速定位** | 哪个项目有风险、哪个 Agent 挂了 | 哪一步失败了、为什么、怎么修 |
| **4. 资源负载可见** | Agent 忙闲程度、Token 消耗趋势 | 当前并发任务数、排队长度 |
| **5. 上下文可追溯** | 项目历史、版本对比 | 每次 Agent 调用的输入输出、审批记录 |
| **6. 人机协作节点** | 哪些项目需要人工审批 | 哪些步骤卡在审批、审批了多久 |

---

## 六、下一步建议

1. **项目图谱**优先做：这是 Dashboard 的延伸，用户每天打开第一眼要看的东西
2. **任务图谱**先做横向流程图（Flow View）：最直观，最容易建立信任
3. **风险节点**用红色脉动动画：B 端用户最怕"不知道出问题了"
4. **产物节点**一定要可点击预览：让用户感受到 Agent 确实在产出价值

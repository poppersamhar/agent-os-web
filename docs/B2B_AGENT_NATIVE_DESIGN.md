# B 端 Agent Native 产品设计方法论

> 基于 Anthropic Managed Agents、多智能体架构调研（LangGraph / CrewAI / AutoGen / MetaGPT / OpenAI Swarm）以及 AgentOS 前端实践的产品设计思考。

---

## 一、Agent Native 的本质：三种产品形态的对比


| 形态               | 代表                | 用户心智                        | 核心问题                      |
| ---------------- | ----------------- | --------------------------- | ------------------------- |
| **Traditional**  | 传统 ERP/CRM        | "我点按钮完成功能"                  | 功能固化，无法适配多变的业务场景          |
| **AI Native**    | Notion AI、Copilot | "我在现有工作流里用 AI 提效"           | AI 是外挂，不改变底层流程            |
| **Agent Native** | 理想的 AgentOS       | "我描述目标，Agent 自主拆解、调度、执行、汇报" | **产品本身没有固定功能，只有可编排的能力单元** |


**关键区别**：

- AI Native：用户用 **功能**，AI 辅助
- Agent Native：用户用 **目标**，Agent 自动组合功能

---

## 二、B 端 Agent Native 的 6 条核心设计原则

### 原则 1：目标驱动，而非流程驱动

**传统 SaaS**：把最佳实践固化成固定流程（审批流 → 配置 → 提交）
**Agent Native**：用户描述目标，Agent 动态生成最优执行路径

```
传统： "点击'创建报表' → 选择模板 → 配置字段 → 生成"
Agent Native： "帮我分析 Q3 各业务线营收，突出异常项" 
              → Agent 自动：提取数据 → 计算同比 → 生成图表 → 标注异常 → 推送结论
```

**设计要点**：

- 前端保留"快捷指令"（降低表达成本），但底层不固化流程
- 每个任务执行后，把实际路径沉淀为"可复用模板"，供下次调用

---

### 原则 2：三层解耦——治理层、调度层、执行层必须独立

这是 Anthropic Managed Agents 调研中最重要的启发：


| 层级                     | 职责                      | 对应产品模块                 |
| ---------------------- | ----------------------- | ---------------------- |
| **治理层（Governance）**    | 审计、风控、合规、效能度量           | 效能仪表盘、审计日志、风险告警        |
| **调度层（Orchestration）** | 理解目标、拆解任务、分配 Agent、整合结果 | 群主 Agent、任务流可视化、上下文管理  |
| **执行层（Execution）**     | 具体干活：查数据、写代码、发邮件、调 API  | Agent 广场、Skill 市场、沙箱执行 |


**设计要点**：

- 治理层必须能**穿透查看**调度层和执行层的全量操作记录
- 调度层必须**无状态**（挂掉可恢复），所有状态存在持久化日志中
- 执行层必须**可替换**（一个 Skill 挂了，换另一个同类 Skill 继续）

---

### 原则 3：上下文是产品，不是技术细节

B 端最大的痛点不是"Agent 不够聪明"，而是 **"这次任务的上次上下文在哪？"**

**Agent Native 的设计**：

```
┌─────────────────────────────────────────┐
│  项目 Session（群聊）                     │
│  ├── 人类输入的历史                       │
│  ├── 群主Agent 的执行计划                 │
│  ├── 各 Agent 的 tool_call + result      │
│  ├── 上传的文件和生成的产物               │
│  └── 审批节点的人类决策记录               │
│                                          │
│  这些不是"聊天记录"，而是                 │
│  可查询、可切片、可审计的企业资产          │
└─────────────────────────────────────────┘
```

**设计要点**：

- Session Log 是产品的核心数据资产，不是临时缓存
- 支持跨 Session 检索历史（"上次类似的项目是怎么做的？"）
- 上下文管理对用户可见（右侧显示"当前加载了 12 个事件 / 32K tokens"）

---

### 原则 4：可观测性即产品功能

B 端客户买的不只是"结果"，更是"过程可控"。

**Agent Native 必须把可观测性做成交互**：


| 传统可观测性（埋给工程师的） | Agent Native 可观测性（给用户看的） |
| -------------- | ------------------------ |
| 日志文件           | 右侧审计日志面板（时间线）            |
| 监控大盘           | 任务执行流可视化（横向步骤条）          |
| 错误堆栈           | Agent 失败时的自动降级 + 人工介入提示  |
| 性能指标           | 上下文占用率、Token 消耗、自动化覆盖率   |


**设计要点**：

- 每个任务执行时，用户实时看到"谁在干什么、进度如何"
- 失败时不只报错误，还要显示"备选方案"和"人工介入入口"
- 治理层能离线分析所有 Session，输出风险报告和能力缺口

---

### 原则 5：Human-in-the-Loop 是设计，不是补救

B 端不能搞"黑箱 Agent"，必须在关键节点**显式设计人类审批**：

```
任务流示例：
计划 → [审批] → 数据提取 → [审批] → 生成图表 → 自动完成
              ↑                  ↑
           高敏感操作          高风险输出
```

**设计要点**：

- 审批节点在状态机中**显式定义**，不是临时加的
- 人类审批时，能看到完整的上下文（为什么 Agent 这么做、依据是什么）
- 审批决策本身也要写入 Session Log，作为审计依据

---

### 原则 6：Skill/Agent 是市场，不是配置

B 端企业的需求差异极大，产品不能靠官方团队堆功能。

**Agent Native 的设计**：


| 层面         | 设计                                               |
| ---------- | ------------------------------------------------ |
| **官方层**    | 提供基础 Skill（SQL执行、文件读写、邮件发送）和基础 Agent（数据分析、知识检索）  |
| **第三方层**   | 通过 MCP/A2A 接入外部 Skill/Agent（GitHub、Slack、Jira 等） |
| **企业自定义层** | 企业自己封装内部 Skill（连接内部数据库、调用内部 API）                 |


**设计要点**：

- Skill 要有"应用商店"的体验：搜索、评分、一键安装
- 每个 Skill 有明确的输入输出 Schema、权限范围、成本估算
- 企业可以管控"哪些 Skill 可被哪些角色调用"

## 四、交互层的设计范式（参考 AgentOS 前端实践）

基于 AgentOS 的前端探索，B 端 Agent Native 的交互应该遵循：

### 范式 1：群聊即工作区

- 每个项目是一个"群聊"
- 群主 Agent 是主持人，人类和 AI 员工是群成员
- 所有操作（@调用、文件上传、审批）都在群聊上下文内完成

### 范式 2：可视化是信任的载体

- 右侧永远显示"任务在怎么执行"
- 不是给工程师看的监控，而是给用户看的进度
- 知识图谱、任务流、Agent 状态、审计日志四视图切换

### 范式 3：渐进式自动化

- 新项目默认高人工介入（每个步骤都确认）
- 运行几次后，用户可以选择"记住这个模式，以后自动执行"
- 自动化程度可调节（从"全人工"到"全自动"滑动条）

## 五、与 C 端 Agent 产品的根本区别


| 维度       | C 端 Agent（如豆包、ChatGPT） | B 端 Agent Native（如 AgentOS） |
| -------- | ---------------------- | --------------------------- |
| **核心诉求** | 好用、快、便宜                | 可控、可审计、合规、ROI 可衡量           |
| **容错率**  | 高（说错了用户再试）             | 极低（错误决策可能带来业务损失）            |
| **交互模式** | 一对一对话                  | 多角色群聊 + 可视化面板               |
| **记忆方式** | 用户维度的长期记忆              | 项目维度的结构化 Session Log        |
| **失败处理** | "抱歉，我没理解"              | 自动降级 + 备选方案 + 人工介入          |
| **成本敏感** | 低（个人付费）                | 极高（企业按 token 计费，需优化）        |

## 六、B 端 Agent Native 产品的设计公式

```
B 端 Agent Native 产品 =
    自然语言入口（降低使用门槛）
  + 状态机调度引擎（保证可控性）
  + 持久化 Session（实现可审计）
  + 可视化执行面板（建立信任）
  + 渐进式自动化（匹配企业接受度）
  + 开放 Skill/Agent 市场（满足差异化需求）
  +  Governance 层（满足合规要求）
```

## 七、下一步建议

AgentOS 目前的前端已经**验证了交互形态**，但还没有后端引擎。建议下一步：

1. **用 Python + FastAPI + OpenAI Agents SDK 搭建最小可用的 Harness 后端**
2. **先实现一个真实的"数据分析 → 图表生成"端到端任务**
3. **把 Session Log 落库（SQLite/PostgreSQL），实现审计可追溯**

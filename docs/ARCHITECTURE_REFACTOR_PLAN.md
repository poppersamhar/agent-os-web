# Agent-OS 架构重构：从独立 Agent 到主-从 Agent 模式

> 基于新的主-从 Agent 架构的产品页面重构方案

---

## 一、新架构核心定义

```
人类员工
    │ 下达目标 / 审批
    ▼
┌─────────────────────────────────────┐
│          主 Agent（群主）            │
│  • 项目管理                          │
│  • 任务拆解与分发                     │
│  • 任务审核                          │
│  • 向人类汇报                        │
└─────────────────────────────────────┘
    │ 分发任务
    ▼
┌─────────────────────────────────────┐
│         Sub-Agent 执行队列           │
│  Sub-1 → Sub-2 → Sub-3 → ...       │
│  （主 Agent 的复制品，排队执行）      │
└─────────────────────────────────────┘
    │ 调用
    ▼
┌──────────────┐    ┌──────────────┐
│   Tools      │    │   Skills     │
│  原子操作     │    │  组合能力     │
│  • 数据提取   │    │  • 图表生成   │
│  • SQL执行   │    │  • 报告排版   │
│  • 文件读写   │    │  • 语义搜索   │
│  • 邮件发送   │    │              │
└──────────────┘    └──────────────┘
```

**关键变化**：
- 不再有"独立 Agent"概念
- 只有一个**主 Agent**（项目维度）+ **Sub-Agent 复制品**（任务执行单元）
- Sub-Agent 是执行者，不是有身份的角色
- 主 Agent 是人类与执行层的唯一接口

## 二、页面重构方案

### 2.1 删除：数字员工页（AgentPage）

**原因**：在新架构下，Sub-Agent 是主 Agent 的复制品，不是独立实体。用户不需要"浏览 Agent 列表"，只需要看到"主 Agent 在管理什么"。

### 2.2 新增：「工具与连接」页（Tools & Connect）

这是替换数字员工页的最佳选择。

#### 为什么不是单独的 MCP 页或 Tools 页？

| 方案 | 问题 |
|------|------|
| 单独的 MCP 页 | MCP 只是协议，用户不关心协议，只关心"能连什么" |
| 单独的 Tools 页 | 工具太多太碎，需要搭配外部连接才有意义 |
| **Tools & Connect** | 左边是原子能力，右边是外部接入，逻辑上是一体的："系统能干什么 + 能连什么" |

#### 页面结构

```
┌──────────────────────────────────────────────────────────────┐
│  [工具与连接]                                                │
│                                                              │
│  ┌──────────────┐  ┌──────────────────────────────────────┐ │
│  │  左侧：Tools  │  │           右侧：Connect              │ │
│  │              │  │                                      │ │
│  │ 内置原子工具  │  │  外部服务接入（MCP / API / DB）      │ │
│  │              │  │                                      │ │
│  │ 📊 数据提取   │  │ ┌─────────────────────────────────┐  │ │
│  │ 📝 文件读写   │  │ │ 🔌 已连接（绿色）               │  │ │
│  │ 📧 邮件发送   │  │ │ • GitHub MCP                    │  │ │
│  │ 🗄️ SQL执行   │  │ │ • 企业内部数据库                 │  │ │
│  │ 🔍 网页抓取   │  │ │ • Slack 通知                    │  │ │
│  │ ...          │  │ └─────────────────────────────────┘  │ │
│  │              │  │                                      │ │
│  │ 点击 Tool    │  │ ┌─────────────────────────────────┐  │ │
│  │ 右侧显示     │  │ │ ➕ 可接入（灰色）               │  │ │
│  │ 调用次数     │  │ │ • Jira MCP                      │  │ │
│  │ Token消耗    │  │ │ • Notion API                    │  │ │
│  │ 成功率       │  │ │ • 飞书机器人                    │  │ │
│  └──────────────┘  │ └─────────────────────────────────┘  │ │
│                     └──────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

#### Tools vs Skills vs Connect 的区分

| 概念 | 定位 | 例子 | 用户理解 |
|------|------|------|---------|
| **Tools** | 系统内置的原子操作 | 数据提取、SQL执行、文件读写 | "系统本身会干什么" |
| **Skills** | 组合编排的能力单元 | 图表生成、报告排版、风险评估 | "Agent 掌握了什么技能" |
| **Connect** | 外部系统接入 | GitHub、Slack、Jira、内部数据库 | "能连到外部什么系统" |

**关系链**：
```
Connect（外部系统）→ 提供数据/能力
         ↓
    Tools（原子操作）→ 处理数据
         ↓
    Skills（组合编排）→ 完成业务目标
         ↓
    Sub-Agent（执行单元）→ 调用 Skills
         ↓
    主 Agent（调度者）→ 分配任务、汇报结果
```

```

### 3.2 节点样式

| 节点 | 形状 | 颜色 | 动画 |
|------|------|------|------|
| 人类 | 圆角矩形 | `#2563eb` 蓝 | 静态 |
| 主 Agent | 大圆（64px）| `#1e3a5f` 深蓝 | 脉动（工作中） |
| Sub-Agent | 中圆（36px）| `#0d9488` teal | 状态色：绿/橙/红 |
| Skills | 六边形（28px）| `#7c3aed` 紫 | 调用时闪烁 |
| 产物 | 文件图标 | `#475569` 灰 | 生成时飞入 |

### 3.3 边的含义

```
人类 → 主 Agent：目标指令（实线箭头）
主 Agent → Sub：任务分发（虚线箭头，带任务名标签）
Sub → Skills：工具调用（细虚线）
Sub → 产物：执行结果（点状线）
主 Agent → 人类：汇报（双向，带回执）
```

## 四、群聊逻辑调整

### 4.1 发言者角色

| 角色 | 功能 | 展示样式 |
|------|------|---------|
| **人类** | 下达目标、审批决策 | 灰色圆角气泡 |
| **主 Agent** | 汇报进度、请求审批、展示结果 | 无气泡纯文本 + 头像 |
| **系统消息** | Sub-Agent 开始/完成、异常告警 | 居中灰色小字 |

**Sub-Agent 不直接出现在群聊中**——它的执行过程在右侧「任务流」面板可视化展示。

### 4.2 群聊与任务流的关系

```
┌──────────────────────────────┬──────────────────────┐
│          群聊区域             │      右侧面板        │

│  主Agent：收到，拆解为3个子任务 │  [步骤1] 🎯 目标理解  │

## 五、Skills 页面保留逻辑

**Skills 页面保留，但重新定义**：

```
Skills = Sub-Agent 可调用的组合能力库
```

展示内容：
- Skill 名称 + 描述
- 包含哪些 Tools（原子工具列表）
- 适用场景
- 调用次数、成功率
- 被哪些项目使用过

**与 Tools & Connect 的关系**：
- Tools & Connect："系统有什么能力、能连什么外部系统"
- Skills："Agent 会什么技能、这些技能由什么工具组成"
- 用户先配置 Tools & Connect，再编排成 Skills，最后由 Sub-Agent 调用
